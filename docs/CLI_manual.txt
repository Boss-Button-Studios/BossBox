  BossBox Playtesting Manual

  Prerequisites

  1. Ollama running
  ollama serve   # if not already running as a service

  2. Models pulled (at least one of these)
  ollama pull smollm:1.7b      # default — fast, small
  ollama pull smollm:360m      # even faster, lower quality
  ollama pull qwen2.5-coder:1.5b

  3. BossBox installed
  cd ~/Projects/BBS/BossBox
  source .venv/bin/activate
  pip install -e ".[dev]"   # already done — just activate venv

  ---
  Running BossBox

  The entry point is bossbox. Always activate the venv first.

  source .venv/bin/activate
  bossbox --help

    usage: bossbox [-h] [--auto] [--redirect TEXT] [--model MODEL] [--no-color]
               goal

    BossBox — local-first AI workbench.

    positional arguments:
      goal             Goal for BossBox to accomplish.

    options:
      -h, --help       show this help message and exit
      --auto           Trust Mode: skip the human decomposition checkpoint.
      --redirect TEXT  Apply a redirect at the checkpoint instead of prompting.
      --model MODEL    Ollama model to use (default: smollm:1.7b).
      --no-color       Disable ANSI colour output.


  ---
  Basic Usage

  Trust Mode (no checkpoint)

  bossbox "write a Python function that reverses a string" --auto
  Runs the full pipeline without pausing. Good for quick experiments.

  Interactive Checkpoint Mode (default)

  bossbox "write a Python function that reverses a string"
  The pipeline pauses after decomposition and shows you the plan. You choose:
  - y or yes — proceed
  - n — abort
  - r focus only on the function, skip tests — redirect with new instructions

  Redirect flag (non-interactive redirect)

  bossbox "write a Flask web app" --redirect "focus on just the route handlers, no templates"
  Applies the redirect automatically at the checkpoint without prompting.

  Custom model

  bossbox "explain recursion" --model smollm:360m
  bossbox "explain recursion" --model qwen2.5-coder:1.5b

  No color (plain text)

  bossbox "summarize async/await" --no-color

  Notifications (Step 17)

  BossBox can send notifications when pipeline events occur. Three channels are
  available; OS native is on by default, the others are opt-in.

  Configuration lives in config/notify.yaml. The file ships with all optional
  sections commented out. Uncomment and fill in what you need:

  OS native (default, no setup required)
    Fires a desktop notification via the OS notification system.
    Degrades silently on headless machines — nothing breaks if no display is present.
    notify:
      os_native:
        enabled: true

  ntfy.sh (recommended for background tasks)
    Sends a push notification to a topic you control. Subscribe on your phone
    or any device with the ntfy app or a plain HTTP client.
    Set the topic via environment variable or inline:
      BOSSBOX_NTFY_TOPIC=your-topic-name
    Self-hosted ntfy is supported — set base_url to your server.
    notify:
      ntfy:
        enabled: true
        base_url: https://ntfy.sh
        topic: ${BOSSBOX_NTFY_TOPIC}

  SMTP email (optional)
    Sends a templated email on task complete/failed, and optionally on
    human checkpoints (email_on_checkpoint: true).
    Email content is strictly templated — event type, title, and timestamp only.
    No pipeline output, document content, or model reasoning is ever included.

    Gmail — use an App Password (myaccount.google.com/apppasswords):
      host: smtp.gmail.com  port: 587  use_tls: true

    Proton Mail — requires Proton Mail Bridge (proton.me/mail/bridge) running
    locally. Bridge exposes SMTP at 127.0.0.1:1025. Use your Bridge password
    (not your Proton account password). Set use_tls: false — Bridge handles
    encryption upstream.
      host: 127.0.0.1  port: 1025  use_tls: false

    notify:
      smtp:
        enabled: true
        host: ${BOSSBOX_SMTP_HOST}
        port: 587
        username: ${BOSSBOX_SMTP_USER}
        password: ${BOSSBOX_SMTP_PASSWORD}
        from_address: ${BOSSBOX_SMTP_FROM}
        to_address: ${BOSSBOX_SMTP_TO}
        use_tls: true
        email_on_checkpoint: false

  Event types that can trigger notifications:
    task_complete, task_failed       — all enabled channels
    human_checkpoint                 — OS + ntfy; email only if email_on_checkpoint: true
    injection_warn, injection_block  — OS + ntfy only
    privilege_escalation             — OS + ntfy only
    anomaly_detected                 — OS + ntfy only
    model_update                     — OS only

  ntfy priority levels:
    urgent — injection_block, privilege_escalation
    high   — human_checkpoint, injection_warn, anomaly_detected
    default — task_complete, task_failed, model_update

  ---
  Audit Log

  Every run is logged to ~/.bossbox/audit/audit.log as JSONL.

  tail -20 ~/.bossbox/audit/audit.log | python3 -m json.tool

  Each entry has event_type, task_id, timestamp, and data. You'll see
  stage_transition, checkpoint_skipped/checkpoint_approved, input_shield,
  action_shield, and notify_event entries.

  notify_event entries record the event_type and title only — no credentials,
  no message body, no SMTP passwords appear in the audit trail.

  ---
  Known Limitations at Step 17

  - Notifications are not yet wired into the CLI pipeline — the Notification
    Service (Step 17) is implemented and configurable, but the CLI runner does
    not call it yet. Notification wiring happens when the GUI pipeline is
    assembled (Step 22). The config and channel code are ready; they just have
    no caller in the CLI path.
  - No real security shields yet — PassthroughShield is wired in; the real
    Hypervisor shield comes in Step 21. Every input passes.
  - No document sanitization in the CLI — the sanitizer (Step 9) exists but is
    not wired into the CLI pipeline yet.
  - Decomposition quality depends on the model — smollm:1.7b may produce
    minimal or off-format YAML. If the decomposer can't parse the response, it
    wraps your original goal as a single task and continues.
  - No skill injection — skills (Steps 18–19) are not wired yet; the model
    gets a bare goal prompt.
  - Single model for both decompose and execute — the tiered model fleet
    (Step 20) isn't wired yet.
  - Exit code — bossbox exits 0 on success, 1 on abort or failure. Scripts
    can check $?.
